# P3-3: Stats 中的 pool_in_use 统计偏差

**时间**: 2026-06-05 21:37 UTC
**状态**: ✅ 已修复
**文件**: model.go

## 问题描述

```go
// Put 里
p.inUse.Add(-1)     // 先减
p.resources <- res  // 后归还（可能阻塞在 channel 发送上）
```

导致 Stats 读数出现 `pool_in_use + pool_available < total_size` 的短暂不一致。

## 修复方案

将 `inUse.Add(-1)` 移到 `p.resources <- res` **之后**，确保归还成功再减计数：

```go
func (p *Pool[T]) Put(res *resource[T]) error {
    if res == nil {
        return nil
    }
    if err := p.connControl.Reset(res.Conn); err != nil {
        _ = p.manager.Send(func(a *PoolManagerActor[T], s *PoolManagerState[T]) {
            a.connControl.Close(res.Conn)
            a.poolTotalSize.Add(-1)
        })
        p.inUse.Add(-1)  // Reset 失败，连接被关闭，inUse 要减
        return err
    }

    if p.waitQueue.TryDequeue(res) {
        p.inUse.Add(-1)  // 直接交给等待者
        return nil
    }

    select {
    case p.resources <- res:
        p.inUse.Add(-1)  // 归还成功后再减
        return nil
    default:
    }

    p.inUse.Add(-1)  // channel 满，连接被关闭
    _ = p.manager.Send(func(a *PoolManagerActor[T], s *PoolManagerState[T]) {
        a.connControl.Close(res.Conn)
        a.poolTotalSize.Add(-1)
    })
    return nil
}
```

实际上当前代码已经是每个分支都 `p.inUse.Add(-1)` 了，问题只是把归还成功那一个分支的 `Add(-1)` 从 `<-res` 前移到 `<-res` 后。这样可以确保 `inUse` 递减和 `resources` 递增原子化（从观察者角度）。

- model.go: Put() 方法，调整 Add(-1) 位置
