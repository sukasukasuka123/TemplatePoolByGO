系统工作流：
1. 审查代码 → 发现错误
2. 给出解决方案，每个方案落到 `docs/plan/{错误名}-{时间戳}.md`
3. 检查方案间是否存在冲突
4. 改动代码修复错误
5. 审查测试用例是否健全（功能测试足够？边界测试完善？）
6. 按需求完善测试代码
7. 运行测试
8. 重复 1-7 直到审查无错且测试符合预期
9. 给出最终代码 review 和新 README

**Why:** 确保每次改动有记录、可回溯、经过充分测试验证。

**How to apply:** 每个改动周期都走完整流程，先写 plan 再改代码，改完检查测试。
